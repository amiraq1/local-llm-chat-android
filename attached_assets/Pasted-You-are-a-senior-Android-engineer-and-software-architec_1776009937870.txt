You are a senior Android engineer and software architect.

I have an Android app called `local-llm-chat-android`. The app already uses a clean Android stack with:
- Kotlin
- Jetpack Compose
- MVVM
- Hilt
- Room
- DataStore
- a local inference abstraction centered around `InferenceEngine`
- a concrete implementation like `MLCInferenceEngine`

Your task is to design and implement **Day 1** of a transition from a local chat app into a **local action assistant**.

## Goal
Do NOT build a full AI agent.
Do NOT add Accessibility automation yet.
Do NOT change the existing app architecture unless necessary.

Instead, add a clean and extensible **Tool Calling layer** above the existing `InferenceEngine`.

## What I want built
Implement a minimal but production-style architecture with these concepts:

1. `Tool`
   - a common interface for app tools/actions
   - each tool has:
     - unique name
     - description
     - input schema or clearly typed params
     - suspend execute function
     - structured result

2. `ToolResult`
   - typed result model
   - includes:
     - tool name
     - success/failure
     - result text
     - optional structured payload
     - optional error message

3. `ToolRegistry`
   - central registry that exposes available tools
   - can return tool by name
   - can list all available tools

4. `ActionOrchestrator`
   - receives a user request
   - decides which tool to call using simple deterministic logic for now
   - no LLM-based tool selection yet unless it is optional and clearly isolated
   - returns a final response that can later be shown in UI

5. Initial tools to implement:
   - `GetDeviceInfoTool`
   - `GetClipboardTool`
   - `GetBatteryStatusTool`

## Important constraints
- Preserve clean architecture and separation of concerns.
- Reuse existing dependency injection with Hilt.
- Keep code easy to extend for future tools.
- Avoid overengineering.
- Prefer strongly typed Kotlin models over vague maps where possible.
- If Android permission or platform limitations apply, handle them safely and explicitly.
- Clipboard and battery access should use Android APIs appropriately.
- Device info should return useful but simple data like manufacturer, model, Android version, SDK level.
